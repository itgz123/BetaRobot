```
❯ 怎么保证两次调用DWT_CNT_Update之间的时间间隔不超过一次溢出

● 这是这个模块的一个设计约束，目前没有强制保证机制，依赖的是使用者的调用频率。

  数学上的硬限制

  CYCCNT 是 32 位寄存器，168MHz 下溢出周期约 25.5 秒。如果两次 DWT_CNT_Update() 间隔超过这个时间，光凭 cnt_now < s_cyccnt_last 无法判断溢出了 1 次还是 N 次——这是 32 位计数器扩展成
   64 位的先天缺陷，无法修复。

  当前实际状态：靠使用模式保证

  所有 DWT_GetTimeUs() 的调用者运行频率都很高：

  ┌───────────────────────────────────────────┬────────────┬────────┐
  │                  调用者                   │  典型频率  │  间隔  │
  ├───────────────────────────────────────────┼────────────┼────────┤
  │ 电机反馈 (drv_dmmotor/djimotor)           │ 1kHz       │ 1ms    │
  ├───────────────────────────────────────────┼────────────┼────────┤
  │ APP 任务看门狗 (motor/chassis/cmd/sensor) │ 100Hz~1kHz │ 1~10ms │
  ├───────────────────────────────────────────┼────────────┼────────┤
  │ 编码器测速 (bsp_tim)                      │ 控制循环内 │ <10ms  │
  ├───────────────────────────────────────────┼────────────┼────────┤
  │ Daemon 任务                               │ 1kHz       │ 1ms    │
  ├───────────────────────────────────────────┼────────────┼────────┤
  │ CAN 超时轮询                              │ while 循环 │ <1μs   │
  ├───────────────────────────────────────────┼────────────┼────────┤
  │ BMI088 SPI 时间戳                         │ 1kHz       │ 1ms    │
  └───────────────────────────────────────────┴────────────┴────────┘

  全部远低于 25 秒阈值，实际上不会触发多重溢出。但如果将来有人写了一个低频任务调用 DWT_GetTimeUs()（比如每 30 秒一次），时间轴就会跳变。

  怎么加固

  最稳妥的做法是在 SysTick 中断里定期调用 DWT_CNT_Update()（不一定是完整的 DWT_GetTimeUs，只需要溢出检查）。FreeRTOS 的 1kHz tick 已经足够。可以在 bsp_dwt.c 暴露一个轻量接口：

  // 供 SysTick/Timer ISR 调用的轻量版本，仅做溢出检测
  void DWT_CNT_Update(void);

  然后在 HAL_SYSTICK_Callback() 或 RTOS 的 tick hook 中调用它，保证即使在没有任何用户代码调用 DWT_GetTimeUs() 的极限情况下，溢出也能被记录。
```
